# CodeHelper 题目填充计划

日期：2026-04-07

## 1. 目标

基于当前产品定位，把题库扩展为 5 条训练赛道：

1. 考研复试机试
2. 保研夏令营
3. 算法岗校招
4. 硬件 / IC 校招
5. 数学建模比赛

本轮规划目标：

- 每条赛道可刷题单不少于 200 题
- 每条赛道都具备“基础题 + 专题题 + 模拟题 + 压轴题”的完整结构
- 题目可以被多个赛道复用，但每条赛道对用户可见的题单数必须 `>= 200`
- 题库最终不只是一堆散题，而是能按用途组织成“考研 / 保研 / 就业 / 建模”四类入口

建议总目标：

- 赛道题单总量：`>= 1000`
- 实体题目总量：第一阶段做到 `850 ~ 950`
- 其中允许约 `10% ~ 20%` 的高价值题跨赛道复用

## 2. 核心假设

为了让这个计划能真正落进当前项目，先明确 4 个假设：

1. “每种题目不少于 200 题”指的是每条赛道的可刷题单数量，而不是必须 100% 全部唯一题。
2. 复试、保研、算法岗这三类可以优先按标准 OJ 题来落地，和当前题库结构最兼容。
3. 硬件 / IC 与数学建模不能完全照搬传统 OJ，需要补 `Verilog/仿真题` 和 `数据/案例型题` 两类题目模式。
4. 第一阶段先把“题目入口 + 标签 + 题单”建好，第二阶段再补更强的执行器与评测能力。

## 3. 当前题库现状

当前资源文件统计：

- `basic.json`: 48 题
- `leetcode.json`: 90 题
- `math-modeling.json`: 35 题
- 合计：`173` 题

当前问题：

- 题源维度只有 `builtin / leetcode / math-modeling`
- 没有“赛道”概念，无法直接给用户展示“考研 / 保研 / 就业 / IC / 建模”入口
- 没有 `Verilog` 语言支持
- 没有“数据处理题 / 仿真题 / 论文复现题”的题型区分
- 数学建模题虽然已经有一部分，但数量和题型结构都远远不够

## 4. 题库结构改造建议

### 4.1 新增元数据维度

建议每道题新增以下元数据：

- `tracks`: 所属赛道，可多选
  - `postgrad-retest`
  - `summer-camp`
  - `algo-job`
  - `ic-job`
  - `math-modeling`
- `platform`: 原始来源平台
  - `pat`
  - `pta`
  - `csp`
  - `luogu`
  - `uoj`
  - `kattis`
  - `cf-gym`
  - `nowcoder`
  - `leetcode`
  - `hackerrank`
  - `codesignal`
  - `hdlbits`
  - `eda-playground`
  - `cumcm`
  - `pgmcm`
  - `mcm-icm`
  - `mathorcup`
  - `kaggle`
  - `mathworks`
- `mode`: 题目模式
  - `oj`
  - `simulation`
  - `data-task`
  - `case-study`
  - `report-task`
- `topic`: 专题标签
- `exam_style`: `acm / oa / modeling / hdl`
- `year`: 年份或届次
- `official_url`: 官方链接
- `estimated_time`: 建议完成时长

### 4.2 资源文件拆分建议

建议从现在的 3 个文件，扩展成按赛道 + 平台拆分：

```text
resources/problems/
  exam-retest-pat.json
  exam-retest-pta.json
  exam-retest-csp.json
  summer-kattis.json
  summer-cf-gym.json
  summer-uoj.json
  algo-job-nowcoder.json
  algo-job-leetcode.json
  algo-job-oa.json
  ic-job-hdlbits.json
  ic-job-nowcoder-verilog.json
  ic-job-simulation.json
  modeling-official.json
  modeling-kaggle.json
  modeling-mathworks.json
```

这样做的好处：

- 后面补题不会把一个大文件越堆越乱
- 可以按赛道分批上线
- 更容易做导入、统计和质量验收

## 7. 推荐实施顺序

按当前项目的技术条件，建议这样推进：

### 阶段 A：先做最兼容的三条赛道

顺序：

1. 考研复试机试
2. 算法岗校招
3. 保研夏令营

原因：

- 都可以主要用 OJ 题落地
- 和当前 Python / C / C++ 执行链最兼容
- 能最快做出用户感知最强的题库扩展

### 阶段 B：补硬件 / IC 赛道

前置条件：

- 增加 `Verilog` 语言类型
- 增加 `iverilog` 或同类仿真执行器
- 增加仿真输出展示

### 阶段 C：补数学建模赛道

前置条件：

- 增加 `data-task / case-study / report-task` 三类题型
- 支持题目挂载数据集、附件、参考输出
- 支持“赛题拆解”与“论文复现”型内容

## 8. 赛道内部题单建议

每条赛道建议都不是单一题库，而是做成以下结构：

1. 入门题单：40~60 题
2. 高频专题题单：80~100 题
3. 模拟训练题单：40~50 题
4. 冲刺压轴题单：20~30 题

例如考研复试机试：

- PAT / PTA 基础输入输出与模拟
- 哈希、排序、链表、树
- 图论与最短路
- DP 与搜索
- 复试模拟卷

例如算法岗校招：

- 高频 150
- 公司真题 80
- OA 模拟 30
- 大厂强化 20

## 9. 题目质量验收标准

每道题至少满足：

- 有中文标题
- 有清晰题面
- 有来源平台
- 有赛道标签
- 有专题标签
- 有难度
- 有 3 个以上测试用例
- 有 starter code
- 有标准输出说明

每个赛道至少满足：

- 题量 `>= 200`
- 专题 `>= 10`
- 模拟卷 `>= 5`
- hard 题占比不低于 15%
- 平台来源不低于 3 个
