## 一、产品定位分析

### 1.1 当前定位

CodeHelper 将自己定位为"AI 驱动的桌面编程助手"，核心卖点是将代码编辑器、AI 对话、题库系统、知识库检索与错题追踪整合在同一个 Electron 桌面应用中。

### 1.2 定位诊断

**问题：定位模糊，缺乏聚焦的用户场景。**

"AI 编程助手"是一个极其宽泛的标签。CodeHelper 试图同时覆盖至少三个不同场景：

| 场景         | 核心用户   | 竞品                     |
| ------------ | ---------- | ------------------------ |
| 刷题练算法   | 备考求职者 | LeetCode、牛客           |
| AI 辅助编码  | 日常开发者 | Cursor、GitHub Copilot   |
| 知识管理 RAG | 知识工作者 | Notion AI、Obsidian+插件 |

**致命问题：每个场景都有一个做得更深的竞品。CodeHelper 在每个场景都是"60 分"，而用户会选择"90 分"的专项工具。**

### 1.3 建议定位

聚焦到一个真正有护城河的方向：**"面向编程学习者的 AI 刷题工作站"**。把"刷题 + AI 辅导 + 错题追踪 + 学习统计"做成闭环，这才是 CodeHelper 相对独立编辑器和在线平台的差异化价值。

## 三、用户价值分析

### 3.1 目标用户画像

**主要用户：正在准备编程面试/考试的大学生和初级开发者**

- 年龄：18-28 岁
- 场景：刷力扣、准备春招秋招、PAT/CSP 考试备考
- 痛点：需要在多个平台之间切换（LeetCode 刷题、ChatGPT 问问题、本地编辑器写代码、笔记软件记录错题）
- 期望：一个工具搞定"写代码 + 问 AI + 记错题 + 看进度"

### 3.2 用户为什么会用？

| 动机           | 强度 | 说明                      |
| -------------- | ---- | ------------------------- |
| 一站式体验     | 中   | 不需要在多个工具间切换    |
| AI 辅导解题    | 高   | 遇到不会的题可以直接问 AI |
| 错题自动追踪   | 中   | 不用自己手动记录          |
| 本地离线使用   | 低   | 大多数学生更习惯在线工具  |
| 学习进度可视化 | 中   | 看到自己的进步有正向激励  |

### 3.3 用户为什么会离开？

| 离开原因                 | 概率 | 说明                                         |
| ------------------------ | ---- | -------------------------------------------- |
| 题库太小                 | 极高 | LeetCode 有 3000+ 题，CodeHelper 只有 158 题 |
| 没有社区                 | 极高 | 无法看别人的解法、无法讨论                   |
| AI 需要自己配置 API Key  | 高   | 大部分学生不会或不愿配                       |
| 数据无法同步             | 高   | 换电脑/清缓存数据全丢                        |
| 编辑器不如 VSCode/Cursor | 中   | 日常编码还是会回去用主编辑器                 |

## 四、最大短板

### 4.1 与竞品相比远不如的方面

1. **题库数量和质量（vs LeetCode）**：158 题 vs 3000+ 题。CodeHelper 的题库是静态 JSON，无法与任何 OJ 平台同步。题目缺少官方难度标注、题解、相关题推荐。

2. **AI 代码理解深度（vs Cursor）**：Cursor 有完整的代码库索引、inline 补全、@ 引用文件/函数。CodeHelper 的 AI 是一个独立的聊天窗口，无法感知用户当前正在编辑的代码、正在做的题目上下文。

3. **社区和竞争机制（vs LeetCode/牛客）**：没有排名、没有讨论区、没有题解分享、没有比赛模式。刷题的核心驱动力之一是社区竞争感。

4. **知识库语义检索（vs Obsidian + 向量插件）**：知识库仅有关键词匹配，没有真正的 RAG 能力。向量嵌入字段已预留但从未实现。

5. **数据持久化和同步（vs 任何在线平台）**：统计数据存在 localStorage、题库和配置存在本地 SQLite。没有云同步、没有账号系统。

6. **判题安全性（vs 任何 OJ）**：代码运行器直接 spawn 子进程执行用户代码，没有容器/沙箱隔离（安全审计已指出此问题）。

### 4.2 产品体验短板

1. **无引导/Onboarding**：新用户打开应用后没有任何引导流程，直接看到一个空白的题目列表和一个"欢迎使用 CodeHelper"的编辑器标签页。不会配置 AI 的用户直接废了一半功能。

2. **无截图/截图预览**：README 中所有截图占位符都是注释状态，无法直观了解产品长什么样。

3. **AI 与编辑器割裂**：AI 对话是独立模块，刷题时的 AI 是独立侧边栏。两者无法互通——你不能在编辑器里选中代码然后问 AI。

## 五、最大风险

### 5.1 技术风险

| 风险                  | 严重程度 | 说明                                                                                                                                                           |
| --------------------- | -------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **代码执行无沙箱**    | 严重     | `codeRunner.ts` 直接 spawn 子进程执行用户代码。虽然有超时和输出限制，但没有进程隔离、文件系统隔离、网络隔离。恶意代码可读取/删除用户文件。**这是发布阻塞项。** |
| **依赖 CVE**          | 高       | 安全审计发现 2 个严重 CVE。`pdf-parse`、`better-sqlite3` 等原生模块存在已知漏洞。                                                                              |
| **Electron 版本过新** | 中       | 使用 Electron 41（2026 年最新），但未确认是否经过充分的稳定性测试。                                                                                            |
| **SQLite 无并发保护** | 中       | 多个 IPC handler 共享同一个数据库连接，高并发写入可能死锁。                                                                                                    |

### 5.2 产品风险

| 风险                     | 严重程度 | 说明                                                                                            |
| ------------------------ | -------- | ----------------------------------------------------------------------------------------------- |
| **定位模糊导致无法获客** | 严重     | "AI 编程助手"太泛，在任何用户群体中都无法形成口碑传播。                                         |
| **题库版权风险**         | 高       | 题目来源包括力扣、牛客、PAT 等平台，直接内置这些题目的完整描述和测试用例可能侵犯版权。          |
| **用户留存无闭环**       | 高       | 没有间隔复习、没有学习路径、没有目标设定、没有成就系统。用户刷完 158 题后就没有理由继续使用了。 |
| **API Key 配置门槛**     | 高       | 最有价值的 AI 功能需要用户自行配置 OpenAI API Key。对非技术用户是巨大障碍。                     |

### 5.3 市场风险

| 风险                             | 严重程度 | 说明                                                                                                           |
| -------------------------------- | -------- | -------------------------------------------------------------------------------------------------------------- |
| **Cursor/GitHub Copilot 的碾压** | 严重     | 这些产品有数十亿美元投入、完整的代码理解能力、庞大的用户基础。CodeHelper 试图在"AI 编程"赛道与之竞争毫无胜算。 |
| **LeetCode 的网络效应**          | 高       | LeetCode 有 3000+ 题、排名系统、企业题库、讨论区、竞赛。用户没有理由离开 LeetCode。                            |
| **Electron 应用分发难度**        | 中       | 相比 Web 应用，桌面应用的获客和更新成本更高。                                                                  |

## 六、最有价值功能（用户最可能付费的功能）

按付费意愿排序：

1. **AI 辅导解题（非通用 AI 聊天，而是针对具体题目的渐进式提示）** -- 如果 AI 能做到"不直接给答案，而是像真人教练一样引导思考"，这是有付费价值的。当前实现只是一个通用聊天窗口加了 3 个快捷提示按钮，远未达到此水平。

2. **智能错题分析和复习计划** -- 自动识别薄弱知识点、生成间隔复习计划、在最佳时间点推送错题重做。当前仅有"AI 分析错误"按钮，没有复习计划。

3. **题库和在线评测** -- 如果题库足够大（1000+）、有实时 OJ 体验、支持竞赛模式，有付费基础。当前 158 题完全不够。

4. **学习数据分析仪表板** -- 类似 GitHub Contributions 的学习画像，可用于求职展示。当前统计数据存储在 localStorage，不具备可分享性。

## 七、最缺失功能（没有这个用户会离开）

按紧迫程度排序：

| 优先级 | 功能                                  | 说明                                                                                          |
| ------ | ------------------------------------- | --------------------------------------------------------------------------------------------- |
| P0     | **云账号和数据同步**                  | 用户不能接受换电脑/清缓存后所有学习数据丢失                                                   |
| P0     | **题库扩展至 1000+**                  | 158 题无法支撑任何有意义的刷题需求                                                            |
| P0     | **AI 上下文感知**                     | AI 必须知道用户当前在做哪道题、写了什么代码、历史提交记录。当前 AI 对话完全不知道用户在做什么 |
| P1     | **间隔复习系统（Spaced Repetition）** | 错题本没有复习推送功能，用户不会主动去回顾                                                    |
| P1     | **代码自动补全**                      | Monaco 默认补全很弱，需要接入 AI 做 inline suggestion                                         |

| P2     | **学习路径推荐**                      | 按用户水平和目标岗位推荐刷题顺序                                                              |
| P2     | **代码提交历史和版本对比**            | 每次提交都保存，可以看到同一题不同版本的解法演进                                              |

## 九、商业化阻塞项（不能收费的原因）

1. **没有账号系统**：无法识别用户身份，无法关联付费状态，无法做使用限制。

2. **AI 功能需要用户自带 API Key**：如果用户已经有了 OpenAI API Key，他们可以直接用 ChatGPT/Cursor，没有理由为 CodeHelper 付费。商业化必须提供托管的 AI 服务（即 CodeHelper 自己的 API 代理层）。

3. **题库可能有版权问题**：如果靠题库吸引用户付费，需要确保题目内容的合法授权。

4. **数据无法保值**：用户的学习数据存在本地，无法导出、无法迁移。如果 CodeHelper 停止维护，用户数据全部作废。这让用户不敢投入时间和金钱。

5. **核心功能没有网络效应**：LeetCode 可以用排名、竞赛、讨论区来构建用户粘性。CodeHelper 是纯本地工具，没有任何社交/网络效应。

6. **产品成熟度不够**：当前版本（v1.0.0）实际上是一个原型/MVP，尚未经过真实用户的检验。直接收费会导致口碑崩塌。

## 十一、战略建议

### 11.1 停止做什么

- 停止试图做"通用 AI 编程助手"。这个赛道有 Cursor 和 GitHub Copilot，它们有数十亿美元投入。
- 停止在编辑器功能上与 VSCode 竞争。Monaco 基础功能够了，不需要分屏、终端面板这些 VSCode 已经做得很好的东西。
- 停止堆砌功能（代码片段、命令面板、全局搜索），聚焦核心场景。

### 11.2 开始做什么

1. **聚焦"AI 刷题教练"场景**：把题目 + AI 辅导 + 错题追踪 + 学习统计做成一个完整闭环。这是 CodeHelper 的独特价值。

2. **AI 必须有上下文感知**：当用户在做一道题时，AI 应该自动知道题目描述、用户代码、历史提交、错题记录。这比通用聊天有价值 10 倍。

3. **解决 API Key 问题**：要么提供托管 AI 服务（CodeHelper API），要么集成免费模型（Ollama 一键安装集成），降低使用门槛。

4. **统计数据迁移至 SQLite**：这是技术债，但也是产品可信度的基础。

5. **间隔复习（Spaced Repetition）**：错题本 + 间隔复习是唯一能让用户持续回来的功能。

6. **题库扩充**：至少 500 题才能初步满足刷题需求。考虑接入开源题库或与 OJ 平台 API 集成。

### 11.3 商业化路径建议

| 阶段      | 策略                                                                                |
| --------- | ----------------------------------------------------------------------------------- |
| 现阶段    | 开源免费，获取用户反馈，打磨核心体验                                                |
| 6 个月后  | 推出 CodeHelper Pro：托管 AI 服务 + 更大题库 + 学习路径 + 数据云同步。定价 $5-10/月 |
| 12 个月后 | 推出 CodeHelper Teams：面向培训机构/高校的团队版本，按座位收费                      |
