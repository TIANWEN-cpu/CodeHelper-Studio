## 评分等级定义

| 等级 | 名称             | 含义                                   |
| ---- | ---------------- | -------------------------------------- |
| L1   | Demo             | 基础原型，仅在理想条件下可运行         |
| L2   | MVP              | 核心功能可用，粗糙但可接受             |

## 一、功能完整度

**当前等级：L2 MVP**

**评分依据：**

各功能模块完成度参差不齐，整体处于"框架已搭建但深度不足"的状态：

| 模块              | 完成度 | 等级 |
| ----------------- | ------ | ---- |
| Monaco 代码编辑器 | 75%    | L2   |
| AI 智能对话       | 65%    | L2   |
| 题库系统          | 70%    | L2   |
| 知识库 RAG        | 35%    | L1   |
| 错题本            | 60%    | L2   |
| 统计面板          | 50%    | L2   |
| 全局搜索          | 30%    | L1   |
| 命令面板          | 45%    | L2   |
| 个性化设置        | 55%    | L2   |

核心功能模块（编辑器、AI 对话、题库）均处于 65%-75% 的完成度，基本可用但存在明显短板。知识库 RAG 仅有关键词匹配，向量嵌入字段为空预留（35%）；全局搜索功能极弱（30%）。题库仅 158 题，无法支撑真正的刷题需求。

**达到 L3 所需：**

1. 题库扩充至 500+ 题，并提供题解和相关题推荐（预计 3-4 周）
2. AI 对话实现上下文感知——自动读取当前题目描述、用户代码、历史提交（预计 2-3 周）
3. 知识库实现向量嵌入与语义检索，而非仅关键词匹配（预计 2-3 周）
4. 全局搜索支持全文搜索、结果高亮、搜索历史（预计 1 周）
5. 错题本增加间隔复习（Spaced Repetition）功能（预计 1-2 周）
6. 统计面板数据从 localStorage 迁移至 SQLite，支持目标设定和趋势分析（预计 1-2 周）

**预计工时：10-14 周（1 人全职）**

## 二、用户体验

**当前等级：L1 Demo**

**评分依据：**

- **无新用户引导（Onboarding）**：新用户首次打开应用后直接看到空白题目列表和一个编辑器标签页，没有任何引导流程。不会配置 AI API Key 的用户直接丧失一半功能价值。
- **无产品截图/预览**：README 中所有截图占位符均为注释状态，用户无法预知产品形态。
- **AI 与编辑器割裂**：AI 对话为独立模块，编辑器中选中代码无法直接向 AI 提问，两个核心功能之间缺乏联动。
- **无快捷键系统**：缺少键盘快捷键自定义能力。
- **错误处理已有基础改进**：已实现 ErrorBoundary、ErrorToast、ErrorWithRetry 组件，但覆盖不全面。
- **无障碍性**：有 accessibility.md 文档，但实际实现有限。
- **主题系统**：已支持 3 套主题切换，有基本的深色模式。

应用在理想条件（用户已配置好 API Key、明确知道要做什么）下可以使用，但对新用户极不友好。

**达到 L2 所需：**

1. 实现首次启动引导流程，引导用户配置 API Key 并完成第一道题目（预计 1-2 周）
2. 添加关键路径的 Loading 状态和空状态提示（预计 1 周）
3. 添加核心功能的 Tooltip 和操作提示（预计 3-5 天）
4. AI 对话集成当前编辑器上下文（当前题目、代码）（预计 1-2 周）

**预计工时：4-6 周（1 人全职）**

## 三、技术架构

**当前等级：L2 MVP（偏 L3）**

**评分依据：**

- **进程模型清晰**：采用 Electron 标准的三进程架构（Main / Preload / Renderer），职责分离合理。
- **IPC 通信类型化**：定义了 IPC 通道类型映射，类型安全有基础保障。
- **状态管理合理**：使用 Zustand 进行状态管理，5 个 Store（appStore、chatStore、editorStore、problemStore、settingsStore）职责分明。
- **模块化结构**：src/modules/ 按功能域划分（ai-chat、editor、knowledge、mistakes、problems、search、settings、stats），结构清晰。
- **安全配置到位**：CSP 策略严格、Electron Fuses 已启用、Preload 层做了安全桥接。
- **已知技术债**：tsconfig 严格模式尚未完全开启；存在大量 pending 的重构任务（hooks 提取、类型整合、共享工具函数）；部分 Store 间存在隐式耦合。

架构设计合理，但执行层面的完善度不够——类似一个经过深思熟虑的蓝图，施工只完成了一半。

**达到 L3 所需：**

1. 开启 TypeScript 严格模式并修复所有类型错误（预计 1-2 周）
2. 提取共享 hooks（useAIStream、useCodeExecution）消除组件间重复逻辑（预计 1 周）
3. 整合共享 TypeScript 接口至 src/types/，统一类型定义（预计 1 周）
4. Store selectors 粒度优化，避免不必要的重渲染（预计 1 周）
5. 统一错误处理策略，确保所有异步操作有完善的错误捕获（预计 1 周）

**预计工时：5-7 周（1 人全职）**

## 四、AI 能力

**当前等级：L1 Demo**

**评分依据：**

- **仅支持 OpenAI 兼容 API**：不支持 Anthropic Claude、本地模型（GGUF/Ollama）等其他 Provider。
- **无上下文感知**：AI 对对话完全不知道用户当前在做哪道题、写了什么代码。这是最致命的缺陷——作为"编程助手"，AI 无法理解用户的编程上下文。
- **记忆系统为纯文本匹配**：虽然实现了长期记忆系统，但检索方式为纯文本匹配而非语义向量检索，记忆的召回精度极低。
- **无多模态能力**：不支持图片输入（如截图提问、算法可视化）。
- **RAG 未真正实现**：知识库的 embedding 字段为空预留，仅有关键词匹配。
- **需要用户自带 API Key**：直接与免费 ChatGPT 竞争，用户缺乏为此单独付费的理由。
- **预设提示词有限**：仅 3 个快捷提示按钮，未覆盖渐进式提示（不直接给答案，引导思考）等高级教学模式。

当前 AI 集成本质上是一个通用聊天窗口加上了"代码助手"的标签，缺乏针对编程学习场景的深度定制。

**达到 L2 所需：**

1. 实现 AI 上下文感知：自动将当前题目描述、用户代码、提交历史注入对话上下文（预计 2-3 周）
2. 支持多 Provider（至少增加 Anthropic Claude 和 Ollama 本地模型）（预计 1-2 周）
3. 实现渐进式提示模式：不直接给出完整答案，而是分步引导用户思考（预计 1-2 周）
4. 记忆检索升级为向量嵌入（预计 1-2 周）

**预计工时：5-9 周（1 人全职）**

- **已有 38+ 测试文件**：覆盖 IPC 通信、数据库操作、状态管理、代码执行器、错误处理、工具函数等核心模块。
- **集成测试已建立**：tests/integration/ 下有题目流程、对话流程、编辑器流程、设置流程等集成测试。

## 九、商业化能力

**当前等级：L1 Demo**

**评分依据：**

当前产品在商业化层面几乎为零：

- **无账号系统**：无法识别用户身份，无法关联付费状态，无法做功能限制或用量统计。
- **AI 功能需用户自带 API Key**：这是最核心的商业化阻碍。如果用户已有 OpenAI API Key，他们可以直接使用 ChatGPT 或 Cursor，没有理由为 CodeHelper 付费。
- **无定价策略**：无免费版/付费版的功能分层。
- **无分析埋点**：无法追踪用户行为、功能使用率、留存率等关键商业化指标。
- **题库版权风险**：题目来源包括力扣、牛客、PAT 等平台，直接内置可能侵犯版权，构成商业化法律风险。
- **数据无法保值**：统计数据存在 localStorage，换设备/清缓存全丢。用户不敢投入时间（更不愿投入金钱）。
- **无网络效应**：纯本地工具，无社区、无排名、无竞赛、无分享，缺乏用户粘性闭环。
- **LICENSE 已有**：MIT 许可证已声明，开源合规。

**达到 L2 所需：**

1. 实现基础账号系统（可先用 GitHub/Google OAuth 登录）（预计 3-4 周）
2. 统计数据迁移至 SQLite 并支持云同步（预计 2-3 周）
3. 接入托管 AI 服务（CodeHelper API 代理层）或集成 Ollama 一键安装（预计 2-3 周）
4. 添加基础使用量统计和埋点（预计 1-2 周）
5. 评估题库版权风险，制定合规策略（预计 1 周）

**预计工时：9-13 周（1 人全职）**

## 战略优先级建议

基于评分结果，按投资回报率排序的改进路径：

### 阶段一：稳固 MVP（预计 6-8 周）

优先解决阻塞发布的硬伤和用户体验的基础缺陷：

1. **代码执行沙箱化**（安全审计 BLOCK-1）——发布前必修
2. **修复依赖 CVE**（安全审计 BLOCK-2）——发布前必修
3. **统计数据迁移至 SQLite**（BLOCK-3）——数据丢失风险不可接受
4. **实现新用户引导流程**（BLOCK-4）——没有引导等于没有产品
5. **AI 上下文感知**（当前题目+代码注入对话）——最高价值的功能提升

### 阶段二：走向 Beta（预计 8-12 周）

在 MVP 稳固后，聚焦于产品核心竞争力——"AI 刷题教练"闭环：

1. 题库扩充至 500+，带题解和难度标签
2. 间隔复习系统（Spaced Repetition）
3. 多 AI Provider 支持（Claude、Ollama 本地模型）
4. 渐进式 AI 提示模式（引导思考而非直接给答案）
5. TypeScript 严格模式、Store 测试补全、E2E 测试

### 阶段三：生产就绪（预计 12-16 周）

为正式商业化做准备：

1. 账号系统和云同步
2. 托管 AI 服务（解除 API Key 门槛）
3. 代码签名和正式发布流程
4. 使用量埋点和分析体系
5. 题库版权合规

CodeHelper 在任何单一维度上均未达到竞品水平。但作为开源项目，其文档质量（L3）和架构设计思路（L2 偏 L3）已超出同类个人项目的平均水平。如果收缩定位至"AI 刷题教练"并集中投入 AI 上下文感知和题库扩充，有机会在细分赛道建立差异化优势。

---

> 结论：CodeHelper 当前处于 L2 MVP 阶段——一个工程质量不错的原型，但尚未达到可公开发布的 Beta 水平。最紧迫的升级方向是 AI 能力（L1 -> L2）和用户体验（L1 -> L2），这两项是用户是否愿意继续使用的关键门槛。文档质量（L3）是当前最强项，应保持并作为其他维度追赶的标杆。
